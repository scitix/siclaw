## 2. 故障根因分析

这是一个**复合故障**，根因如下：

### 🔴 直接原因：CoreDNS 环路检测触发 CrashLoopBackOff

从 `coredns-bbbbb` 的日志可以看到关键错误：

```
[FATAL] plugin/loop: Loop (127.0.0.1:46253 -> :53) detected for zone "."
```

`loop` 插件检测到 **DNS 转发环路**——CoreDNS 的 `forward . /etc/resolv.conf` 配置让它读取容器内的 `/etc/resolv.conf`，而该文件指向的 upstream DNS 恰好又回到了 CoreDNS 自身，形成死循环。

### 🔴 次要因素：PDB 锁死 + 节点状态异常

| 问题 | 影响 |
|------|------|
| `PDB: minAvailable=2` | 只允许 0 个中断，导致 1 个 pod crash 后无法自动驱逐重建 |
| `node-a: SchedulingDisabled` | crash 的 pod 卡在不可调度的节点上，无法漂移到其他健康节点 |

### 🔴 业务影响：DNS 查询超时

- kube-dns service 背后应该有两个 endpoint（两个 coredns pod）
- 业务 pod 的 DNS 请求通过 kube-dns service 负载均衡到两个 backend
- **当请求落到 CrashLoopBackOff 的 pod 时，无响应导致超时**

## 3. 修复建议

### 立即修复（恢复 DNS 服务）

**方法一：修复 localhost 转发问题（推荐）**

修改 CoreDNS ConfigMap，将 `forward` 指向明确的 upstream DNS，而非 `/etc/resolv.conf`：

```yaml
# kubectl edit cm coredns -n kube-system
forward . /etc/resolv.conf {
    force_tcp  # 或改为明确的 DNS 如 8.8.8.8、VPC DNS
}
```

或者注释掉 `loop` 插件（如果确定无环路风险）。

**方法二：强制重建 CoreDNS pod**

由于 PDB 限制，需要暂时放宽或手动干预：

```bash
# 临时放宽 PDB（如允许 1 个不可用）
kubectl patch pdb coredns-pdb -n kube-system -p '{"spec":{"minAvailable":1}}'

# 删除 crash 的 pod，让它在新节点重建

# 确认两个 pod 都 Running 后恢复 PDB
kubectl patch pdb coredns-pdb -n kube-system -p '{"spec":{"minAvailable":2}}'
```
